# 运维与发布说明

本文档用于说明 Diana Monitor 的日常运维流程、发布步骤与回滚策略。

## 1. 发布前检查

- 确认环境变量已配置完整（见 `README.md`）
- 语法检查通过：

```bash
node --check server.js
node --check lib/api-heavy.js
node --check assets/js/main-runtime.js
node --check assets/js/tokens.js
```

- 若修改了定时任务相关功能，建议先本地自测：
  - `GET /api/cron-usage`
  - `POST /api/cron-toggle`

## 2. 发布步骤

1. 同步代码到服务器目录（示例：`/home/ubuntu/diana-monitor`）
2. 重启服务：

```bash
systemctl --user daemon-reload
systemctl --user restart diana-monitor
systemctl --user is-active diana-monitor
```

3. 冒烟验证：

```bash
curl -s http://localhost:18790/health
curl -s -H "Authorization: Bearer <token>" "http://localhost:18790/api/bots"
curl -s -H "Authorization: Bearer <token>" "http://localhost:18790/api/cron-usage?bot=diana"
```

## 3. 定时任务相关说明

- 后端通过 OpenClaw CLI 拉取任务：
  - `cron list --all --json`（包含已暂停任务）
- 按钮行为：
  - 暂停：`enabled=0`
  - 恢复：`enabled=1`
- 若点击后前端未更新，通常是缓存窗口导致，等待几秒后自动刷新

## 4. 常见问题

### 4.1 旧任务显示“处理中”

已在前端增加兜底：超过阈值仍未收口的历史任务会标记为“超时未收口”。

### 4.2 暂停后任务消失

已修复：后端改为 `cron list --all`，已暂停任务仍会显示并可恢复。

### 4.3 首页 Unauthorized

当前策略是仅放行 `index.html` 和 `/assets/*`，API 仍强制鉴权。  
若页面脚本提示缺少 token，请使用带 token 的入口链接首次访问。

## 5. 回滚策略

1. 回退到上一个稳定 commit
2. 重启服务
3. 复验 `health`、`cron-usage`、WebSocket 连通性

## 6. 安全建议

- 不要在 `diana-monitor.service` 中保留明文密钥
- 建议使用 `EnvironmentFile=/path/to/.env` 并设置最小文件权限
- 定期轮转：
  - `DIANA_MONITOR_ACCESS_TOKEN`
  - `DIANA_OPENCLAW_TOKEN`
  - Feishu App Secret
  - `OPENAI_API_KEY`
